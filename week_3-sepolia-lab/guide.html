Blockchain Fundamentals

왜 블록체인이 필요한가?

기존 시스템의 한계를 이해하고, 블록체인이 그 문제를 어떻게 해결하는지 알아봅니다.

🏦

중앙화 시스템

은행, 정부, 대기업이 모든 기록을 관리

문제점

  • • 조작 가능 — 관리자가 기록을 임의로 변경할 수 있다
  • • 단일 장애점 — 서버 하나가 다운되면 전체 시스템 마비
  • • 높은 수수료 — 중개자가 수수료를 가져간다
🌐

탈중앙화 시스템

수천 대의 컴퓨터가 동일한 기록을 보관

장점

  • • 조작 불가 — 과반수 이상을 동시에 속여야 하므로 사실상 불가능
  • • 장애 내성 — 일부 노드가 꺼져도 나머지가 계속 작동
  • • 중개자 없음 — 직접 거래하므로 수수료 절감

💡 비유로 이해하기 — 마을 장부

마을에 이장이 혼자 장부를 관리한다고 해보자.
이장이 몰래 장부를 고치면? 아무도 모른다. → 조작 가능

이제 마을 사람 전원이 같은 장부를 한 부씩 갖고 있다고 해보자.
누군가 자기 장부를 고치면? 나머지 999명의 장부와 다르니까 바로 들통난다. → 조작 불가능

이것이 바로 블록체인이다.

해시(Hash) — 데이터의 지문

어떤 데이터를 넣든 고정 길이의 고유한 값이 나온다.
1글자만 바꿔도 완전히 다른 값이 나온다 — 이를 눈사태 효과(Avalanche Effect)라고 한다.

// 입력이 조금만 달라도 해시 값이 완전히 다르다 "Hello" → 0x185f8db3... "Hello!" → 0x334d016f...

왜 중요한가?

블록체인에서 각 블록은 이전 블록의 해시를 포함한다.
블록 하나를 조작하면 → 해시가 바뀌고 → 그 뒤의 모든 블록도 바뀌어야 한다 → 사실상 조작 불가능.

1. 코인과 단위 기초

지갑은 돈을 담는 곳이 아니다

흔한 오해

"지갑에 코인이 들어있다" → 틀림!

실제

코인은 블록체인(공유 장부)에 기록되어 있다. 지갑은 그 기록에 접근하는 열쇠일 뿐!

💡 비유 — 금고 열쇠

금(코인)은 금고(블록체인)에 있고, 지갑은 금고 열쇠일 뿐이다.
열쇠를 잃어버리면 금에 접근할 수 없고, 열쇠를 남에게 주면 금을 도둑맞는다.

🔑

개인키 (Private Key)

절대 남에게 보여주면 안 되는 비밀번호

64자리 16진수

📢

공개키 (Public Key)

개인키에서 수학적으로 파생된 공개 정보

128자리 16진수

📬

주소 (Address)

0x로 시작하는 42자리 — 다른 사람에게 알려줘도 됨

0x + 40자리 16진수

💡 비유로 이해하기

개인키 = 도장 원판 (절대 남에게 보여주면 안 됨)
공개키 = 도장 자국 (도장을 찍으면 누구나 볼 수 있음)
주소 = 사서함 번호 (누구나 편지를 보낼 수 있음)

wei / gwei / ether 단위 체계

1m = 100cm = 1,000mm인 것처럼, 이더리움에도 단위 체계가 있다. (1BTC의 최소 단위가 사토시(satoshi)인 것처럼)

단위 값 비유 주요 용도
wei 1 1원 내부 연산
gwei 109 wei 1,000원 Gas 가격
ether (ETH) 1018 wei 1조 원 일반 거래

왜 이렇게 작은 단위가 필요한가?

  • ▶ 이더리움은 소수점 이하 18자리까지 표현 가능
  • ▶ Gas 수수료가 매우 작을 수 있으므로 (0.000000001 ETH 등) 작은 단위가 필수
  • ▶ 코드에서는 항상 정수(BigInt)로 계산 → 소수점 오차 방지

4. L2란 무엇인가? 중급

왜 L2가 필요한가?

L1(이더리움 메인넷)은 느리고(12초/블록), 비싸고(수수료 $10~$50), 처리량이 한정(15 TPS)되어 있다.
L2는 이 문제를 해결한다: 빠르고(~2초), 저렴하고(1/100 수수료), 높은 처리량을 제공한다.

💡 비유로 이해하기 — 본점과 지점

L1 = 본점 은행 — 안전하지만 느리고 비쌈. 모든 거래를 직접 처리한다.
L2 = 지점 은행 — 빠르고 저렴. 대부분의 업무를 처리하고 결과만 본점에 보고한다.
GIWA Sepolia = 지점 은행 중 하나 — OP Stack 기반으로 운영되는 L2 지점이다.

Optimistic Rollup이란?

GIWA는 OP Stack 기반 = Optimistic Rollup이다.
"Optimistic(낙관적)"이라는 뜻: 트랜잭션이 유효하다고 가정하고, 문제가 있을 때만 이의 제기(Fraud Proof)를 한다.

💡 비유 — 무인 매장

고객을 믿고 결제를 맡기되, CCTV로 사후에 확인한다.
문제가 발견되면 7일 이내에 이의 제기가 가능하다.
대부분의 경우 문제가 없으므로, 빠르고 효율적으로 처리된다.

5. Nonce 심층 중급

Nonce란?

Nonce(Number used ONCE)는 계정 단위 트랜잭션 카운터다. 0부터 시작하여 트랜잭션을 보낼 때마다 1씩 증가한다.

  • • 이더리움 노드는 nonce가 현재 카운터와 정확히 일치할 때만 트랜잭션을 처리한다
  • • 리플레이 방지: 같은 nonce의 트랜잭션은 한 번만 실행된다
  • • 순서 보장: nonce 3은 nonce 2가 처리된 후에만 실행된다

💡 비유로 이해하기 — 수표 일련번호

은행에서 수표에 일련번호를 매기는 것과 같다.
수표 #0, #1, #2… 순서대로 발행하며, 같은 번호의 수표는 한 번만 결제된다.
누군가 수표 #1을 복사해서 두 번 결제하려 해도 은행은 "이미 처리된 번호"라고 거부한다.

코드에서 Nonce가 나타나는 곳

// 01-check-balance.js — 현재 nonce 조회 var nonce = await provider.getTransactionCount(address); // → 이 주소에서 보낸 트랜잭션 수 = 현재 nonce // 02-send-eth.js — 송금 전후 nonce 비교 var nonceBefore = await provider.getTransactionCount(myAddress); // ... sendTransaction 실행 ... var nonceAfter = await provider.getTransactionCount(myAddress); // nonceAfter === nonceBefore + 1 ← nonce가 1 증가!

6. Gas 심층 중급

Gas란?

Gas는 EVM(이더리움 가상 머신)의 연산 비용 단위다. 모든 작업에는 정해진 gas 비용이 있으며, 사용자는 이 비용을 ETH로 지불한다.

실제 수수료 = gasUsed × gasPrice

단순 ETH 전송: 항상 21,000 gas (고정) · Gas Price: 네트워크 혼잡도에 따라 변동

개념 타입 위치 설명
gasLimit BigInt tx.gasLimit 최대 사용 가능 gas
gasUsed BigInt receipt.gasUsed 실제 사용된 gas
gasPrice BigInt receipt.gasPrice gas 1단위당 가격 (wei)

💡 비유로 이해하기 — 자동차 연료

gasLimit = 연료 탱크 용량 (최대 넣을 수 있는 양)
gasUsed = 실제 소비된 연료 (주행 후 확인)
gasPrice = 주유 단가 (리터당 가격)
수수료 = 실제 소비 연료 × 단가 = 주유비 청구서

인터랙티브: Gas 수수료 계산기

0.00042 ETH

420,000,000,000,000 wei

공식: gasUsed × gasPrice(Gwei) × 109 = 수수료(wei) · 수수료(wei) ÷ 1018 = 수수료(ETH)

7. Signature 심층 중급

ECDSA 디지털 서명

이더리움은 secp256k1 타원곡선을 사용한 ECDSA(Elliptic Curve Digital Signature Algorithm)로 트랜잭션을 서명한다. 서명 결과는 세 값 (r, s, v)로 표현된다.

값 크기 의미
r 32바이트 (64자리 16진수) 타원곡선 점의 x좌표
s 32바이트 (64자리 16진수) 서명 스칼라 값
v 1바이트 복원 파라미터 (0 또는 1)

💡 비유로 이해하기 — 위조불가 인감도장

서명은 "위조 불가 도장"과 같다.
r, s = 도장 무늬 (고유한 패턴)
v = 도장을 찍은 방향
ecrecover = 도장 자국만 보고 "누구 도장인지" 역추적하는 감정사
Private Key(도장 원판)는 네트워크에 전혀 전송되지 않는다!

ecrecover 검증 플로우

8. 코드 실습 1 기초

01-check-balance.js node scripts/01-check-balance.js
Nonce 조회 잔액 확인 Provider 생성
const dotenv = require('dotenv'); const ethers = require('ethers'); dotenv.config(); const RPC_URL = 'https://sepolia-rpc.giwa.io/'; const provider = new ethers.JsonRpcProvider(RPC_URL); const wallet = new ethers.Wallet(privateKey, provider); const balanceWei = await provider.getBalance(address); const balanceEth = ethers.formatEther(balanceWei); const nonce = await provider.getTransactionCount(address);

1. Provider 생성 — 블록체인 연결

const provider = new ethers.JsonRpcProvider(RPC_URL); const wallet = new ethers.Wallet(privateKey, provider);

JsonRpcProvider는 GIWA Sepolia RPC를 통해 GIWA Sepolia에 읽기 전용으로 연결한다. Wallet에 개인키와 provider를 넘기면 서명(쓰기)도 가능해진다.

💡 비유로 이해하기

Provider는 은행 창구 전화번호. Wallet은 계좌번호 + 도장. 전화(Provider)를 걸어 도장(Wallet)으로 본인 확인 후 거래한다.

2. 잔액 + Nonce 조회

const balanceWei = await provider.getBalance(address); const balanceEth = ethers.formatEther(balanceWei); const nonce = await provider.getTransactionCount(address);

getBalance()는 wei(정수) 반환 → formatEther()로 ETH 문자열 변환. getTransactionCount()는 해당 주소가 보낸 TX 수 = 현재 nonce.

💡 비유로 이해하기

getBalance는 통장 잔액 조회, getTransactionCount는 거래 횟수 조회. 1 ETH = 1018 wei이므로 모든 내부 연산은 정수(BigInt)로 처리된다.

기대 출력

Terminal
$ node scripts/01-check-balance.js

[연결 정보]
RPC URL : https://sepolia-rpc.giwa.io/
네트워크 : GIWA Sepolia 테스트넷

[지갑 정보]
주소 : 0x742d35Cc6634C0532925a3b844Bc9e7595f2bD18
잔액 : 0.5 ETH
현재 Nonce: 3

✅ Faucet에서 받은 ETH가 코드로 확인됨

학습 포인트

  • • JsonRpcProvider로 블록체인에 연결 (서버/CLI 양쪽에서 동일)
  • • 잔액은 wei(BigInt)로 저장됨 → formatEther()로 변환
  • • Nonce = 보낸 TX 수. 다음 TX는 이 nonce 번호로 발행됨

Nonce 변화 Gas 계산 TX 전송
const provider = new ethers.JsonRpcProvider(RPC_URL); const wallet = new ethers.Wallet(privateKey, provider); const nonceBefore = await provider.getTransactionCount(myAddress); const tx = await wallet.sendTransaction({ to: toAddress, value: ethers.parseEther(amount), }); const receipt = await tx.wait(); const gasUsed = receipt.gasUsed; const gasPrice = receipt.gasPrice; const feeWei = gasUsed * gasPrice; const feeEth = ethers.formatEther(feeWei);

1. 주소 유효성 검사

try { toAddress = ethers.getAddress(toAddress); } catch (e) { console.log('❌ 잘못된 이더리움 주소:', toAddress); process.exit(1); }

ethers.getAddress()는 EIP-55 체크섬 주소로 정규화하면서 유효성 검사를 수행한다. 잘못된 주소면 예외가 발생한다.

2. 트랜잭션 전송

const tx = await wallet.sendTransaction({ to: toAddress, value: ethers.parseEther(amount), });

sendTransaction은 내부적으로 ① nonce 조회 → ② gas 추정 → ③ ECDSA 서명 → ④ eth_sendRawTransaction 브로드캐스트를 순서대로 실행한다.

💡 비유로 이해하기

은행 창구에 "이 금액을 저 계좌로 이체해줘"라고 맡기는 것. ethers.js가 모든 서류 작성(nonce, gas, 서명)을 자동 처리한다.

3. 영수증 대기 + Gas 수수료 계산

const receipt = await tx.wait(); // 블록 포함 대기 const gasUsed = receipt.gasUsed; // 실제 사용 gas const gasPrice = receipt.gasPrice; // gas 단가 (wei) const feeWei = gasUsed * gasPrice; // BigInt 곱셈 const feeEth = ethers.formatEther(feeWei); // wei → ETH

tx.wait()은 eth_getTransactionReceipt를 폴링하여 블록에 포함될 때까지 대기한다. 영수증에서 실제 가스 사용량과 가격을 읽어 수수료를 계산한다.

💡 비유로 이해하기

tx.wait()은 "이체 완료" 문자가 올 때까지 기다리는 것. 문자(영수증)가 오면 실제 배송비(gas 수수료)를 확인할 수 있다.

4. Nonce +1 증가 확인

const nonceAfter = await provider.getTransactionCount(myAddress); console.log('Nonce:', nonceBefore, '→', nonceAfter, '로 +1 증가!');

💡 비유로 이해하기

수표 일련번호가 3 → 4로 증가했다. 다음 수표는 반드시 #4번이어야 한다.

Terminal

학습 포인트

  • • sendTransaction()이 nonce/gas/서명을 자동 처리
  • • tx.wait()으로 블록 포함 대기 → 영수증 획득
  • • Nonce가 자동으로 +1 증가 (순서 보장)
  • • Gas가 실제로 차감됨 (gasUsed × gasPrice)

10. 코드 실습 3 중급

03-track-tx.js node scripts/03-track-tx.js <TX_HASH>
Nonce 확인 Gas 확인 서명 r,s,v
const provider = new ethers.JsonRpcProvider(RPC_URL); const tx = await provider.getTransaction(txHash); const receipt = await provider.getTransactionReceipt(txHash); console.log('Nonce:', tx.nonce); console.log('r:', tx.signature.r); console.log('s:', tx.signature.s); console.log('v:', tx.signature.v); const feeWei = receipt.gasUsed * receipt.gasPrice;

1. 트랜잭션 + 영수증 조회

const tx = await provider.getTransaction(txHash); const receipt = await provider.getTransactionReceipt(txHash);

getTransaction()은 브로드캐스트 시점의 데이터(TransactionResponse)를 반환한다. getTransactionReceipt()는 블록 포함 후의 실행 결과(TransactionReceipt)를 반환한다.

💡 비유로 이해하기

getTransaction()은 택배 조회(발송 직후 정보), getTransactionReceipt()는 배달 완료 확인서(수령 후 최종 정보).

2. 서명 정보 접근 (r, s, v)

console.log('r :', tx.signature ? tx.signature.r : 'N/A'); console.log('s :', tx.signature ? tx.signature.s : 'N/A'); console.log('v :', tx.signature ? tx.signature.v : 'N/A');

ethers.js v6의 Signature 객체에서 r/s(64자리 16진수)와 v(복원 파라미터)에 접근한다. 이 값들이 ECDSA 서명의 핵심이다.

💡 비유로 이해하기

택배 송장에 찍힌 인감도장 무늬(r, s)와 방향(v)을 확인하는 것. 이 도장 자국으로 보낸 사람을 특정할 수 있다.

3. 수수료 계산

const gasUsed = receipt.gasUsed; const gasPrice = receipt.gasPrice; const feeWei = gasUsed * gasPrice; // BigInt 곱셈 const feeEth = ethers.formatEther(feeWei);

학습 포인트

  • • TransactionResponse와 TransactionReceipt는 다른 객체
  • • 서명 (r, s, v)은 tx.signature에서 접근
  • • 하나의 TX에서 Nonce + Gas + Signature 세 개념이 모두 확인됨

11. 코드 실습 4 고급

04-verify-sig.js node scripts/04-verify-sig.js <TX_HASH>
ecrecover 서명 검증 보너스 실습
const tx = await provider.getTransaction(txHash); // Transaction 객체 재구성 const rawTx = new ethers.Transaction(); rawTx.type = tx.type; rawTx.to = tx.to; rawTx.nonce = tx.nonce; rawTx.gasLimit = tx.gasLimit; rawTx.data = tx.data; rawTx.value = tx.value; rawTx.chainId = tx.chainId; rawTx.signature = tx.signature; if (tx.maxFeePerGas) rawTx.maxFeePerGas = tx.maxFeePerGas; if (tx.maxPriorityFeePerGas) rawTx.maxPriorityFeePerGas = tx.maxPriorityFeePerGas; // .from 접근 시 ecrecover 실행 recoveredAddress = rawTx.from;

1. Transaction 객체 재구성

const rawTx = new ethers.Transaction(); rawTx.type = tx.type; rawTx.to = tx.to; rawTx.nonce = tx.nonce; rawTx.gasLimit = tx.gasLimit; rawTx.value = tx.value; rawTx.chainId = tx.chainId; rawTx.signature = tx.signature;

provider.getTransaction()은 TransactionResponse를 반환하는데, 이는 serialized 속성이 없어 직접 ecrecover를 호출할 수 없다. new ethers.Transaction()으로 직접 재구성하면 .from 접근 시 내부적으로 ecrecover가 실행된다.

2. ecrecover — 서명자 주소 복원

// .from 접근 시 내부적으로 ecrecover 수행 recoveredAddress = rawTx.from;

rawTx.from에 접근하면 ethers.js가 내부적으로 서명(r, s, v) + 트랜잭션 해시 → 타원곡선 역산 → Public Key 복원 → keccak256 해시 → 이더리움 주소로 변환한다.

💡 비유로 이해하기

ecrecover는 "봉인된 편지의 도장 자국"만 보고 누가 찍었는지 역산하는 것. Private Key(도장 원판) 없이도 서명자를 특정할 수 있다.

3. 주소 비교 — 검증 결과

const originalAddress = tx.from.toLowerCase(); const recoveredLower = recoveredAddress.toLowerCase(); if (originalAddress === recoveredLower) { console.log('✅ 서명 검증 성공!'); console.log('→ Private Key를 공개하지 않고도 본인임을 증명!'); }

복원된 주소와 원본 tx.from이 일치하면 수학적 증명 완료. 핵심: Private Key는 네트워크에 전혀 전송되지 않는다.

기대 출력

Terminal
$ node scripts/04-verify-sig.js 0x9f8e...

[서명 데이터]
원본 From : 0x742d...bD18
r : 0x8a7f...3e21
s : 0x4b2c...9d07
v : 0

[검증 결과]
원본 From : 0x742d...bD18
복원 주소 : 0x742d...bD18

✅ 서명 검증 성공!
→ Private Key를 공개하지 않고도 본인임을 증명 가능!

학습 포인트

  • • TransactionResponse → Transaction 재구성이 필요한 이유 이해
  • • .from 접근 = ecrecover 실행 (서명자 주소 복원)
  • • EIP-1559 TX는 maxFeePerGas / maxPriorityFeePerGas 필수
  • • Trustless: 중앙 서버 없이 서명만으로 신원 증명

12. 보안 주의사항 필수

초보자가 자주 하는 실수 TOP 5

1. 개인키를 코드에 직접 입력 (하드코딩)

코드에 개인키를 직접 넣으면 GitHub 등에 노출될 위험이 크다.

❌ 잘못된 예

const key = "0xac09...";

✅ 올바른 예

const key = process.env.PRIVATE_KEY;

2. .env 파일을 git에 커밋

.gitignore에 .env를 반드시 추가해야 한다. 한 번이라도 커밋되면 git 히스토리에 영원히 남는다.

# .gitignore 예시 .env .env.local node_modules/

3. 테스트넷 키를 메인넷에서도 사용

반드시 분리해야 한다! 테스트넷에는 테스트넷 전용 계정을 사용하고, 메인넷 계정과 절대 같은 개인키를 쓰지 않는다.

4. 낯선 사이트에 개인키 입력 (피싱)

어떤 사이트도 개인키를 요구하면 100% 사기이다. MetaMask와 같은 공식 지갑만 사용하고, 의심스러운 링크는 절대 클릭하지 않는다.

5. 서명 요청 내용을 확인하지 않고 승인

MetaMask 팝업에서 항상 내용을 확인해야 한다. 어떤 컨트랙트와 상호작용하는지, 얼마의 금액이 전송되는지 꼭 읽어본 뒤 승인한다.

.env 파일 설정

개인키와 RPC URL은 .env 파일에 저장하고, dotenv 패키지로 불러온다.

# .env 파일 예시 PRIVATE_KEY=0x여기에_개인키_입력 RPC_URL=https://sepolia-rpc.giwa.io/

🔑 개인키 비유

개인키 = 집 열쇠. 누구에게도 복사본을 주지 마세요.
잃어버리면 집(자산)에 다시는 들어갈 수 없습니다.
누군가에게 주면, 그 사람이 내 집(자산)을 마음대로 쓸 수 있습니다.

3대 개념 요약

🔢

Nonce

계정 단위 트랜잭션 카운터

  • • 0부터 시작, TX마다 +1
  • • 리플레이 공격 방지
  • • 처리 순서 보장
provider.getTransactionCount()
⛽

Gas

EVM 연산 비용 단위

  • • ETH 전송 = 21,000 gas 고정
  • • 수수료 = gasUsed x gasPrice
  • • 스팸 방지 + 자원 배분
receipt.gasUsed * receipt.gasPrice
✏️

Signature

ECDSA secp256k1 서명

  • • (r, s, v)로 구성
  • • ecrecover로 서명자 복원
  • • Private Key 미전송 = 보안
tx.signature.r / .s / .v
